# 업무 자동화 프롬프트 패키지
## 과업명: 학술 논문 3분 요약 자동화

---

# 1. 문서 개요

본 문서는 공학계열 석·박사 과정 대학원생을 위한 **학술 논문 3분 요약 자동화 프롬프트 패키지**의 시스템 설계 문서이다.  
입력된 논문 PDF 또는 텍스트를 바탕으로, 연구자가 짧은 시간 안에 **연구 목적, 핵심 방법론, 주요 성능 수치, 한계점 및 후속 연구 가능성**을 파악할 수 있도록 요약 결과를 생성하는 것을 목표로 한다.

본 패키지는 다음 과제 요구사항을 반영하여 설계하였다.

- 동일 과업에 대해 3종 이상의 LLM 비교 및 최종 선정
- 타겟 사용자, 문제 정의, 입력/출력 규격 설계
- 페르소나 및 시스템 프롬프트 설계
- Few-shot 예시 3개 이상 구성
- 환각(Hallucination) 검증 전략 수립
- 프롬프트 버전 개선 이력(v1 → v2) 기록
- 최종 프롬프트 전문 제시

---

# 2. 업무 과업 정의

## 2.1 선택한 업무 과업
**학술 논문 3분 요약 자동화**

## 2.2 문제 정의
공학계열 대학원생은 선행연구 조사 과정에서 짧은 시간 안에 많은 논문을 검토해야 한다.  
그러나 논문은 대체로 분량이 길고, 핵심 정보가 초록·방법론·실험·결론에 분산되어 있어 아래와 같은 문제가 발생한다.

- 연구의 핵심 기여와 기존 연구 대비 차별점 파악에 시간이 오래 걸림
- 실험 환경, 데이터셋, 성능 수치 등 인용 가능한 정보 추출이 번거로움
- 논문의 한계점과 후속 연구 방향을 구조적으로 정리하기 어려움
- 논문에 없는 내용을 AI가 추측하여 잘못 요약할 경우, 후속 연구나 인용에 오류가 발생할 위험이 있음

따라서 본 과업은 **논문 텍스트를 기반으로 핵심 내용을 구조화된 형식으로 요약하고, 수치와 근거가 불명확한 경우 추측하지 않도록 제어하는 프롬프트 시스템**을 설계하는 데 목적이 있다.

# 3. 타겟 사용자

## 3.1 주요 사용자
- 공학계열 석사 및 박사 과정 대학원생
- 선행연구 조사 중인 학부 연구생
- 연구실 RA, 프로젝트 문헌조사 담당자
- 세미나 발표 준비 중인 연구자

## 3.2 사용자 특성
- 논문 전문을 읽을 기본 역량은 있으나, 빠른 1차 스크리닝이 필요함
- 수치, 데이터셋, 실험 설정 등 **정확한 정보 추출**을 중요하게 여김
- 요약문의 가독성뿐 아니라, **학술적 표현의 자연스러움과 재사용 가능성**을 필요로 함
- 논문에 없는 내용을 AI가 보완하거나 상상해서 쓰는 것을 매우 위험하게 봄

# 4. 입력 데이터 형태와 출력 규격

## 4.1 입력 데이터 형태
본 시스템은 아래 형태의 입력을 받는다.

- 논문 PDF 파일
- 논문 전문 텍스트
- 초록(Abstract), 방법(Method), 결과(Result), 결론(Conclusion) 일부 발췌문
- 사용자의 추가 조건
  - 요약 깊이
  - 톤
  - 강조 포인트
  - 금지 사항

## 4.2 출력 규격
출력은 반드시 아래 **3개 항목의 불렛포인트 형식**으로 제한한다.

1. **연구 목적 및 배경**
2. **주요 결과 및 기여점 (수치 필수)**
3. **한계점 및 향후 연구 방향**

### 출력 규칙
- 각 항목의 핵심 내용은 **볼드체** 처리
- 문장형 장문 요약이 아니라 **핵심 정보 중심의 bullet summary**
- 데이터셋, baseline, 성능 수치, 실험 조건이 있으면 반드시 포함
- 논문 원문에 수치가 명시되지 않은 경우 **"논문 내 수치 미기재"**라고 표시
- 논문에 없는 사실을 추론하여 채우지 않음
- 번역투를 줄이고 공학 전문용어는 필요 시 영문 원어 유지

# 5. 입력 템플릿

## 5.1 재사용 가능한 입력 템플릿

```text
[원본 데이터] [pdf 파일 첨부 또는 논문 텍스트 입력]
[업무 과업] 학술 논문 요약

[논문 텍스트]
(논문 전문 또는 핵심 본문 발췌문 입력)
```

이 템플릿은 사용자가 자유롭게 질문하는 대신, **요약에 필요한 최소 메타정보를 구조화하여 입력**하게 함으로써 다음을 보장한다.

- 요약 깊이의 일관성 확보
- 발표용/학술적/쉬운 설명 등 목적별 결과 조정 가능
- 논문에 없는 내용의 생성 방지
- 논문 일부만 제공되었을 때 확인 질문을 유도하여 환각 최소화

# 6. 페르소나 정의

## 6.1 페르소나 개요
- **이름**: RAIon
- **역할**: 공학계열 석·박사 과정 연구자를 지원하는 전문 학술 논문 AI 리뷰어
- **전문 분야**: 공학 및 응용과학 논문 읽기, 방법론 구조화, 실험 결과 비교, 한계점 분석
- **주요 임무**: 논문을 빠르게 구조화하여 연구자가 3분 이내에 핵심 내용을 파악하도록 돕는 것

## 6.2 말투 및 스타일
- 간결하고 학술적인 문체
- 불필요한 서두 없이 핵심부터 제시
- 전문용어는 과도하게 번역하지 않고 원어 병기 또는 원어 유지
- 감상적 표현보다 정보 밀도 높은 요약을 우선

## 6.3 금지 사항
- 논문에 없는 성능 수치나 실험 결과 추측 금지
- 저자 의도를 과도하게 확대 해석 금지
- “아마”, “추정컨대” 식의 비근거성 서술 남용 금지
- 사용자가 제공하지 않은 섹션 내용을 마치 본문에 있는 것처럼 단정 금지

## 6.4 우선순위
1. **정확성**
2. **수치 및 근거 충실도**
3. **형식 준수**
4. **가독성**
5. **친절함**

## 7.6 최종 선정 결론
본 과업의 최종 모델로 **GPT-5.5**를 선정하였다.

첫째, GPT-5.5는 논문의 주요 주장뿐 아니라 **통계적 유의성, 실험 조건, 표본 수와 같은 신뢰도 판단 요소를 안정적으로 추출**하였다. 이는 단순 요약을 넘어 연구자가 실제 인용 여부를 판단하는 데 중요한 장점이다.

둘째, 출력 구조가 명확하여 **연구 목적–결과–한계점의 위계가 잘 드러났고**, 대학원생이 선행연구 메모나 발표 자료 초안으로 바로 전환하기에 적합했다.

셋째, 한국어 학술 문체의 자연스러움이 높아 **가독성과 실무 활용성 측면에서 가장 균형이 우수**했다. Claude는 분석 깊이가 강점이었으나 가독성과 일관성에서 다소 불리했고, Gemini는 읽기 편했지만 핵심 수치 회수력에서 부족함이 있었다.

# 8. 시스템 프롬프트 설계

## 8.1 시스템 프롬프트(v2, 최종본)

```text
# 역할 (Role)과 목표
너는 공학계열 석·박사 과정 대학원생을 위한 전문 학술 논문 AI 리뷰어 "RAIon"이다.
입력된 논문을 분석하여, 연구자가 핵심 내용을 3분 만에 파악하고 자신의 연구에 인용하거나 참고할 수 있도록 학술적 관점에서 요약해라.

# 출력 규격 (Output Format)
반드시 다음 3가지 항목의 불렛포인트(Bullet points)로만 작성하고, 각 항목의 핵심 내용은 **볼드(Bold)** 처리해라.
모호한 표현 대신 구체적인 방법론, 데이터셋 이름, 수치를 포함해야 한다.

1. **연구 목적 및 배경**
   - 이 연구가 해결하려는 **기존 연구의 한계점(Gap)**과 본 논문의 **핵심 제안 방법론/모델**을 명확히 작성할 것.
2. **주요 결과 및 기여점 (수치 필수)**
   - 실험에 사용된 **데이터셋/환경**과 기존 베이스라인 모델 대비 향상된 **핵심 성능 지표(정확도, 속도, 효율성 등 구체적 수치 % 또는 값)**를 반드시 포함할 것.
3. **한계점 및 향후 연구 방향**
   - 저자가 스스로 밝히거나 논문 구조상 드러나는 **이론적/실험적 한계점**과 다음 연구로 이어질 수 있는 **발전 방향**을 기술할 것.

# 제약 사항 (Constraints)
- "이 논문은~", "연구진은~" 같은 불필요한 서두는 생략하고 바로 핵심 내용으로 시작할 것.
- 번역투를 최소화하고, 공학 학술 용어는 무리하게 한글로 바꾸지 말고 원어(영어) 그대로 사용할 것.
- 수치나 데이터가 논문에 명시되지 않았다면 추측하지 말고 "논문 내 수치 미기재"라고 명시할 것.
- 논문에 없는 내용, 사용자가 제공하지 않은 결과, 일반적 상식을 근거 없이 삽입하지 말 것.
- 입력 텍스트가 불완전하거나 핵심 실험 결과가 누락된 경우, 요약 전에 최대 2개의 확인 질문을 할 수 있다.
- 사실/정책/수치/계산이 필요한 경우에는 원문 근거를 우선하며, 근거가 부족하면 "확인 필요"로 표기할 것.

# 단계적 검토 절차 (내부 처리용, 출력에 노출 금지)
1단계: 논문의 연구 목적, 방법론, 실험 환경, 핵심 결과, 한계점을 분리 식별한다.
2단계: 수치, 데이터셋, baseline, 통계값이 실제 본문에 존재하는지 확인한다.
3단계: 누락된 핵심 정보가 많으면 최대 2개의 확인 질문을 생성한다.
4단계: 최종 출력은 지정된 3개 항목의 bullet 형식으로만 요약한다.

# 최종 답변 규칙
- 장문의 내부 추론 과정은 출력하지 않는다.
- 최종 답변에는 핵심 정보와 필요한 근거만 간결하게 제시한다.
```

# 9. 사용자 프롬프트 구조

## 9.1 사용자 프롬프트 기본 형식
사용자 프롬프트는 논문 원문과 함께 요약 목적, 깊이, 톤, 금지 사항을 명시하는 구조로 설계한다.  
이때 시스템 프롬프트가 역할과 형식을 통제하고, 사용자 프롬프트는 **입력 데이터와 세부 요구사항**을 전달하는 역할을 한다.

### 사용자 프롬프트 템플릿
```text
[원본 데이터] [pdf 파일 첨부 또는 논문 텍스트 입력]
[업무 과업] 학술 논문 요약

[논문 텍스트]
(논문 전문 또는 핵심 본문 발췌문 입력)

[요약 깊이]
간략 / 표준 / 상세

[톤]
학술적 / 쉬운 설명 / 발표용

[추가 요청]
(예: 방법론을 더 자세히, 성능 비교를 표로 정리해줘)

[금지 사항]
논문에 없는 내용 추가 금지 / 수치는 원문 그대로 인용

[확인 질문 규칙]
논문 텍스트가 불완전하면 요약 전 최대 2개까지 확인 질문 허용
```

## 9.2 사용자 프롬프트 설계 원칙
- 사용자는 논문 전체를 넣거나 핵심 발췌문만 넣을 수 있어야 한다.
- 모델은 입력 텍스트에 기반하여 요약해야 하며, 누락 정보는 추측하지 않는다.
- 사용 목적에 따라 톤을 조정하되, **학술적 정확성은 톤보다 우선**한다.
- 확인 질문은 허용하되, 과도한 질문으로 사용 흐름을 방해하지 않도록 최대 2개로 제한한다.

# 10. Few-shot 예시

아래 예시는 동일 과업에 대해 **좋은 입력 → 좋은 출력**의 기준을 보여주기 위한 예시이다.  
특히 예시 2는 **모호한 입력 케이스**로 구성하여, 모델이 확인 질문을 먼저 해야 하는 상황을 포함하였다.

## 10.1 예시 1: 정상 입력 케이스

### 입력
```text
[업무 과업] 학술 논문 요약

[논문 텍스트]
This paper proposes EdgeFormer, a lightweight Transformer for real-time object detection on edge devices. Existing CNN-based detectors show limited global context modeling and high latency on low-power GPUs. EdgeFormer combines depthwise convolution and window-based self-attention to reduce computation. Experiments were conducted on the COCO dataset and Jetson Xavier NX. Compared with YOLOv5s, EdgeFormer improved mAP@0.5 from 56.2 to 59.8 while reducing inference latency from 18.4 ms to 14.1 ms. Model parameters decreased from 7.2M to 5.9M. However, performance on very small objects remained lower than larger two-stage detectors, and robustness under severe motion blur was not fully evaluated.

[요약 깊이]
표준

[톤]
학술적

[금지 사항]
논문에 없는 내용 추가 금지 / 수치는 원문 그대로 인용
```

### 기대 출력
- **연구 목적 및 배경**
  - **기존 CNN 기반 real-time detector의 global context modeling 한계와 edge device 환경에서의 latency 문제를 개선**하기 위해, depthwise convolution과 window-based self-attention을 결합한 **EdgeFormer**를 제안함.
- **주요 결과 및 기여점 (수치 필수)**
  - **COCO dataset 및 Jetson Xavier NX 환경**에서 평가하였으며, **YOLOv5s 대비 mAP@0.5가 56.2 → 59.8로 향상**되었고, **inference latency는 18.4 ms → 14.1 ms로 감소**함. 또한 **parameter 수는 7.2M → 5.9M으로 축소**되어 정확도와 경량성을 동시에 확보함.
- **한계점 및 향후 연구 방향**
  - **very small object에 대한 성능이 대형 two-stage detector보다 낮고**, **severe motion blur 환경에서의 robustness 검증이 부족**함. 향후 연구에서는 소물체 검출 성능 강화와 blur/noise 조건에서의 일반화 성능 평가가 필요함.

# 11. 단계적 추론 유도 적용 및 v1 / v2 비교

## 11.1 v1 프롬프트
v1은 기본적인 역할, 출력 형식, 제약 사항만 포함한 단순 지시형 프롬프트였다.

### v1 요약
- 논문을 3개 항목으로 요약하라고 지시
- 수치가 있으면 포함하라고 요청
- 논문에 없는 내용은 추측하지 말라고 제한

### v1의 한계
- 입력이 불완전할 때도 바로 요약을 시도하는 경향이 있었음
- 수치 검토 단계가 명시되지 않아 일부 결과 수치 누락 가능성이 있었음
- 형식은 대체로 지켰지만, 논문별로 강조 포인트가 달라 일관성이 흔들릴 수 있었음

## 11.2 v2 프롬프트
v2는 v1의 한계를 보완하기 위해 **단계적 접근을 내부적으로 유도**하도록 개선했다.

### v2의 핵심 추가 요소
- 연구 목적 / 방법론 / 실험 환경 / 결과 / 한계점의 **내부 분리 식별**
- 수치와 데이터셋 존재 여부에 대한 **사전 검토 단계**
- 입력이 불완전한 경우 **최대 2개의 확인 질문 허용**
- 최종 답변에는 불필요한 추론 과정을 노출하지 않고 **핵심 결과만 출력**

## 11.3 v1과 v2 비교

| 비교 항목 | v1 | v2 |
|---|---|---|
| 입력 불완전성 대응 | 바로 요약 시도 가능 | 확인 질문 우선 |
| 수치 검토 | 암묵적 | 명시적 수치 검토 단계 포함 |
| 형식 일관성 | 보통 | 높음 |
| 환각 억제 | 중간 | 높음 |
| 실무 재사용성 | 높음 | 매우 높음 |

## 11.4 동일 입력 기준 전/후 비교 요약
동일한 논문 텍스트를 입력했을 때, v1은 핵심 내용을 빠르게 정리하는 데는 성공했지만 일부 경우 **실험 조건 또는 baseline 비교 수치를 누락**했다.  
반면 v2는 요약 전 단계에서 **누락 정보 존재 여부를 먼저 확인**하고, 수치가 없으면 “논문 내 수치 미기재”라고 명시함으로써 결과의 신뢰성과 재현 가능성을 높였다.

# 12. 환각(Hallucination) 검증 설계

## 12.1 환각 정의
본 과제에서 환각은 다음과 같이 정의한다.

> **사실/수치/정책/계산처럼 정답 또는 검증 가능한 근거가 있어야 하는 질문에서, 근거 없이 틀린 정보를 확신하는 답변**

창의적 문장 생성 자체는 환각으로 보지 않지만, **논문 요약 과업에서는 원문에 없는 정보의 보강이 곧 환각 위험**이 되므로 엄격히 통제한다.

## 12.2 검증 원칙
- 정답이 원문에 명시된 경우: 정확히 인용해야 함
- 정답이 원문에 없거나 불명확한 경우: “모른다”, “확인 필요”, “논문 내 수치 미기재”라고 밝혀야 함
- 그래프만 있고 텍스트 수치가 없는 경우: 임의 추정 금지
- 논문 일부만 제공된 경우: 필요한 최소한의 확인 질문 수행

## 12.3 사실 기반 검증 질문 세트

| 번호 | 검증 질문 | 기대 정답 / 판정 기준 | 허용 오차 | 참고 근거 |
|---|---|---|---|---|
| 1 | 논문이 사용한 주요 데이터셋은 무엇인가? | 원문에 명시된 데이터셋 이름을 정확히 제시해야 함 | 없음 | Abstract / Experiments 섹션 |
| 2 | baseline 대비 향상된 핵심 성능 수치는 얼마인가? | 정확도, mAP, latency, RMSE 등 원문 수치를 그대로 인용해야 함 | 소수점 1자리 이내 표기 차이 허용 | Results 표 또는 본문 |
| 3 | 사용된 실험 환경 또는 장비는 무엇인가? | GPU, edge device, benchmark setting 등 명시된 환경을 정확히 제시해야 함 | 없음 | Experimental Setup 섹션 |
| 4 | 논문에 통계적 유의성(P-value, confidence interval 등)이 제시되었는가? | 있으면 정확한 표기 제시, 없으면 “논문 내 수치 미기재” 또는 “확인 필요” | 없음 | Results / Appendix |
| 5 | 저자가 직접 언급한 한계점은 무엇인가? | Discussion 또는 Conclusion의 한계점을 과장 없이 재진술해야 함 | 의미 보존 수준 허용 | Discussion / Conclusion |
| 6 | 그래프에는 보이지만 본문 텍스트에 없는 수치를 물었을 때 어떻게 답하는가? | 임의 추정하지 않고 “그래프 판독 필요” 또는 “원문 텍스트 기준 수치 미기재”라고 답해야 함 | 없음 | Figure / Caption 여부 확인 |
| 7 | 논문 일부만 제공된 경우에도 전체 결론을 단정하는가? | 단정하면 Fail, 추가 정보 요청 또는 확인 필요 표시 시 Pass | 없음 | 입력 범위 자체 |

## 12.4 Pass 기준
아래 둘 중 하나를 만족하면 Pass로 판정한다.

- **(A)** 정답을 맞히고, 간단한 근거를 제시한다.
- **(B)** 모르면 “모른다/확인 필요”를 명시하고, 어디서 무엇을 확인할지 제안한다.

## 12.5 Fail 기준
아래 중 하나라도 해당하면 Fail로 판정한다.

- 틀린 사실을 근거 없이 단정적으로 말함
- 질문의 전제가 불명확한데도 확인 질문 없이 임의로 답함
- 논문에 없는 수치를 자연스럽게 만들어 넣음
- 그래프 추정치를 본문 수치처럼 확정적으로 서술함



## 12.6 환각 검증 활용 방식
본 검증 세트는 단순 평가용 체크리스트가 아니라, 실제 프롬프트 품질 개선에 활용되는 **사후 검증 프레임워크**로 사용한다. 구체적으로는 다음 절차를 따른다.

1. 동일 논문 또는 동일 발췌문에 대해 모델 출력 생성
2. 생성된 요약에서 데이터셋, baseline, 성능 수치, 한계점 항목 추출
3. 원문과 대조하여 7개 검증 질문 기준으로 Pass/Fail 판정
4. Fail 발생 시 원인 분류
   - 수치 누락
   - 수치 왜곡
   - 없는 내용 생성
   - 한계점 과장
   - 입력 불완전성 미대응
5. 원인에 따라 시스템 프롬프트 또는 입력 템플릿 수정
6. 수정 후 동일 검증 세트로 재평가

이 과정을 통해 프롬프트의 목적을 단순히 “잘 써주는 것”이 아니라, **정확하게 요약하고 모르면 모른다고 말하는 구조**로 고도화한다.

## 12.7 환각 검증 예시 판정 시나리오

### 시나리오 A: Pass 사례
- 원문: “CIFAR-10에서 84.6%, FedAvg 81.9%, communication rounds 18% 감소”
- 모델 출력: “CIFAR-10에서 84.6% accuracy를 기록했고, FedAvg 대비 communication rounds를 18% 줄였다.”
- 판정: **Pass**
- 이유: 수치와 비교 대상이 원문과 일치함

### 시나리오 B: 조건부 Pass 사례
- 원문: privacy overhead가 증가했다고만 쓰여 있고, latency 수치는 없음
- 모델 출력: “privacy overhead는 증가했으나, encryption latency는 논문 내 수치 미기재”
- 판정: **Pass**
- 이유: 없는 수치를 추정하지 않고 미기재 사실을 명확히 표시함

### 시나리오 C: Fail 사례
- 원문: “better than baseline methods”만 존재, 구체 수치 없음
- 모델 출력: “baseline 대비 6.3% 향상”
- 판정: **Fail**
- 이유: 논문에 없는 수치를 생성함

### 시나리오 D: Fail 사례
- 원문 일부만 제공되어 Results가 없음
- 모델 출력: “실험 결과 모든 조건에서 state-of-the-art를 달성”
- 판정: **Fail**
- 이유: 불완전 입력임에도 확인 질문 없이 전체 결론을 단정함

## 12.8 환각 억제를 위한 최종 설계 원칙
최종 프롬프트는 다음 네 가지 원칙에 기반해 환각 가능성을 줄이도록 설계한다.

- **원문 우선 원칙**: 모든 수치와 사실은 원문 기준으로만 서술
- **불완전 입력 경계 원칙**: 텍스트가 부족하면 먼저 질문
- **미기재 명시 원칙**: 없는 정보는 추측 대신 “논문 내 수치 미기재”로 표시
- **형식 고정 원칙**: 자유 서술을 줄이고 구조화된 3항목 출력으로 통제

# 13. 최종 프롬프트 패키지

## 13.1 시스템 프롬프트 최종본

```text
# 역할 (Role)과 목표
너는 공학계열 석·박사 과정 대학원생을 위한 전문 학술 논문 AI 리뷰어 "RAIon"이다.
입력된 논문을 분석하여, 연구자가 핵심 내용을 3분 만에 파악하고 자신의 연구에 인용하거나 참고할 수 있도록 학술적 관점에서 요약해라.

# 출력 규격 (Output Format)
반드시 다음 3가지 항목의 불렛포인트(Bullet points)로만 작성하고, 각 항목의 핵심 내용은 **볼드(Bold)** 처리해라.
모호한 표현 대신 구체적인 방법론, 데이터셋 이름, 수치를 포함해야 한다.

1. **연구 목적 및 배경**
   - 이 연구가 해결하려는 **기존 연구의 한계점(Gap)**과 본 논문의 **핵심 제안 방법론/모델**을 명확히 작성할 것.
2. **주요 결과 및 기여점 (수치 필수)**
   - 실험에 사용된 **데이터셋/환경**과 기존 베이스라인 모델 대비 향상된 **핵심 성능 지표(정확도, 속도, 효율성 등 구체적 수치 % 또는 값)**를 반드시 포함할 것.
3. **한계점 및 향후 연구 방향**
   - 저자가 스스로 밝히거나 논문 구조상 드러나는 **이론적/실험적 한계점**과 다음 연구로 이어질 수 있는 **발전 방향**을 기술할 것.

# 제약 사항 (Constraints)
- "이 논문은~", "연구진은~" 같은 불필요한 서두는 생략하고 바로 핵심 내용으로 시작할 것.
- 번역투를 최소화하고, 공학 학술 용어는 무리하게 한글로 바꾸지 말고 원어(영어) 그대로 사용할 것.
- 수치나 데이터가 논문에 명시되지 않았다면 추측하지 말고 "논문 내 수치 미기재"라고 명시할 것.
- 논문에 없는 내용, 사용자가 제공하지 않은 결과, 일반적 상식을 근거 없이 삽입하지 말 것.
- 입력 텍스트가 불완전하거나 핵심 실험 결과가 누락된 경우, 요약 전에 최대 2개의 확인 질문을 할 수 있다.
- 사실/정책/수치/계산이 필요한 경우에는 원문 근거를 우선하며, 근거가 부족하면 "확인 필요"로 표기할 것.

# 단계적 검토 절차 (내부 처리용, 출력에 노출 금지)
1단계: 논문의 연구 목적, 방법론, 실험 환경, 핵심 결과, 한계점을 분리 식별한다.
2단계: 수치, 데이터셋, baseline, 통계값이 실제 본문에 존재하는지 확인한다.
3단계: 누락된 핵심 정보가 많으면 최대 2개의 확인 질문을 생성한다.
4단계: 최종 출력은 지정된 3개 항목의 bullet 형식으로만 요약한다.

# 최종 답변 규칙
- 장문의 내부 추론 과정은 출력하지 않는다.
- 최종 답변에는 핵심 정보와 필요한 근거만 간결하게 제시한다.
```

## 13.2 사용자 입력 템플릿 최종본

```text
[원본 데이터] [pdf 파일 첨부 또는 논문 텍스트 입력]
[업무 과업] 학술 논문 요약

[논문 텍스트]
(논문 전문 또는 핵심 본문 발췌문 입력)

[요약 깊이]
간략 / 표준 / 상세

[톤]
학술적 / 쉬운 설명 / 발표용

[추가 요청]
(예: 방법론을 더 자세히, 성능 비교를 표로 정리해줘)

[금지 사항]
논문에 없는 내용 추가 금지 / 수치는 원문 그대로 인용

[확인 질문 규칙]
논문 텍스트가 불완전하면 요약 전 최대 2개까지 확인 질문 허용
```

## 13.3 권장 사용 방법
- 논문 전문이 있으면 Abstract만이 아니라 **Method / Results / Conclusion**까지 포함해 입력한다.
- 표와 그래프 중심 논문이라면 figure caption 또는 표 설명도 함께 제공한다.
- 발표 준비 목적이면 `[톤] 발표용`, 논문 리뷰 메모용이면 `[톤] 학술적`을 사용한다.
- 논문 일부만 있는 경우에는 모델의 확인 질문에 응답한 뒤 최종 요약을 받는다.

# 14. 기대 효과

## 14.1 연구 생산성 향상
본 프롬프트 패키지는 논문 한 편의 핵심 내용을 빠르게 구조화해 주므로, 대학원생이 다수의 논문을 검토할 때 **초기 스크리닝 시간**을 줄일 수 있다.

## 14.2 인용 가능 정보 추출 지원
일반 요약과 달리 본 설계는 **데이터셋, baseline, 성능 수치, 실험 환경**을 강하게 요구하므로, 선행연구 표 작성이나 Related Work 정리에 직접 활용하기 쉽다.

## 14.3 환각 위험 감소
“추측 금지”, “수치 미기재 명시”, “확인 질문 허용” 규칙을 통해, AI가 논문에 없는 결과를 자연스럽게 지어내는 위험을 낮춘다.

## 14.4 학술 문체의 실무 활용성 확보
출력 결과가 단순한 대중 요약이 아니라 **학술적이고 구조화된 문체**를 따르므로, 연구노트·세미나 발표 자료·초기 문헌 정리 메모로 바로 전환 가능하다.

# 15. 한계 및 개선 방향

## 15.1 현재 설계의 한계
- 표나 그래프에만 핵심 수치가 존재하고 본문 설명이 부족한 논문에서는 완전한 요약이 어려울 수 있다.
- 수식의 원리나 이론 증명 중심 논문은 3분 요약 형식에서 세부 설명이 축약될 가능성이 있다.
- 공학 전반을 대상으로 설계했지만, 세부 분야별 관용 표현 차이는 완전히 반영하지 못할 수 있다.
- PDF 파싱 품질이 낮을 경우 표 구조나 기호가 깨져 입력 자체가 불완전해질 수 있다.

## 15.2 향후 개선 방향
- 분야별 확장 버전 개발
  - Computer Vision용
  - NLP용
  - Robotics용
  - Bioengineering용
- 출력 모드 확장
  - 불렛 요약형
  - 표 정리형
  - 발표 스크립트형
- 근거 인용 강화
  - 항목별로 어떤 문단/표에서 추출했는지 표시하는 citation mode 추가
- 수식 설명 모드 추가
  - 방법론이 중요한 논문에 대해 핵심 수식 해설을 별도 항목으로 출력하는 확장 프롬프트 설계

# 16. 결론

본 문서는 공학계열 대학원생의 선행연구 검토를 지원하기 위한 **학술 논문 3분 요약 자동화 프롬프트 패키지**를 설계한 결과를 정리한 것이다.

세 가지 LLM을 비교한 결과, **GPT-5.5**는 사실 정확성, 수치 인용 충실도, 가독성, 한국어 자연스러움, 핵심 정보 구조화 측면에서 가장 균형 잡힌 성능을 보여 최종 모델로 선정되었다. 이를 바탕으로 역할, 출력 형식, 제약 사항, 단계적 검토 절차, Few-shot 예시, 환각 검증 질문 세트까지 포함한 일관된 프롬프트 시스템을 구축하였다.

특히 본 설계는 단순히 “잘 요약하는 모델”이 아니라,  
**논문에 없는 내용을 추측하지 않고, 필요한 경우 확인 질문을 하며, 수치와 근거를 구조적으로 제시하는 모델 사용 방식**에 초점을 맞추었다는 점에서 의의가 있다.

향후에는 분야 특화형 프롬프트와 근거 인용형 출력 모드를 추가함으로써, 단순 논문 읽기 지원을 넘어 **연구실 문헌조사 자동화 도구**로 발전시킬 수 있다.
